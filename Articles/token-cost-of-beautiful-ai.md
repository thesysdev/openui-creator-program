# The Token Cost of Beautiful AI: OpenUI Lang vs. AI SDK vs. JSON — What You're Actually Paying For

> Written for [thesysdev/openui-creator-program](https://github.com/thesysdev/openui-creator-program) — Issue #4: Token Efficiency Analysis

---

## The Hidden Bill You're Not Looking At

Generative UI is everywhere. You prompt an LLM, it spits out a component tree, your framework renders it into a working interface. It feels like magic — until you look at the invoice.

Here's the thing most teams discover too late: the model's response is the single largest token consumer in a Generative UI pipeline. Not the system prompt. Not the user query. The **output**. Every JSON brace, every `"type"` key, every `"props"` wrapper is a token you're paying for at both generation time (output tokens) and decoding time (input tokens on the next turn). Multiply that by millions of requests and you're not paying for intelligence — you're paying for syntax.

This article is a data-driven comparison of four serialization formats used in Generative UI systems: **OpenUI Lang**, **YAML**, **Vercel AI SDK's JSON-Render**, and **Thesys C1 JSON**. All benchmark numbers come from OpenUI's official benchmark, generated by GPT-5.2 at temperature 0, measured with the `tiktoken` `gpt-5` encoder.

The goal isn't to tell you what to use. It's to show you exactly what each approach costs — in tokens, in dollars, and in milliseconds — so you can make an informed decision.

---

## The Four Contenders

Before the numbers, let's define what we're comparing. Each format represents the same logical component tree — a set of UI elements with types, properties, and hierarchical structure. They just express it differently.

### 1. YAML (The Status Quo)

YAML has been the default for AI-generated configuration for years. It's human-readable, indentation-based, and widely supported. But for Generative UI, it carries significant token overhead.

```yaml
root: stack-1
elements:
  textcontent-2:
    type: TextContent
    props:
      text: "Employees"
      variant: "large-heavy"
  table-3:
    type: Table
    props:
      columns:
        - key: name
          label: "Name"
          type: "string"
        - key: role
          label: "Role"
          type: "string"
      rows:
        - { name: "Alice", role: "Engineer" }
        - { name: "Bob", role: "Designer" }
```

**316 tokens** for a simple table. Every component needs a `type` key, a `props` wrapper, and nested indentation. The structure is explicit but verbose.

### 2. Vercel AI SDK JSON-Render

Vercel's AI SDK uses JSON Patch operations to stream incremental UI updates. Each element is added via an `op: "add"` operation with a path.

```json
{"op":"add","path":"/root","value":"stack-1"}
{"op":"add","path":"/elements/textcontent-2","value":{"type":"TextContent","props":{"text":"Employees","variant":"large-heavy"}}}
{"op":"add","path":"/elements/table-3","value":{"type":"Table","props":{"columns":[{"key":"name","label":"Name","type":"string"},{"key":"role","label":"Role","type":"string"}],"rows":[{"name":"Alice","role":"Engineer"},{"name":"Bob","role":"Designer"}]}}}
```

**340 tokens**. The JSON Patch format adds operational overhead — each element requires an `"op"`, `"path"`, and `"value"` wrapper on top of the standard JSON structure. The benefit is native streaming support. The cost is more tokens per element.

### 3. Thesys C1 JSON

C1 JSON is a component-centric JSON format where each element is wrapped in a `{"component": {...}}` envelope.

```json
{"component":{"component":"Stack","props":{"children":[{"component":{"component":"TextContent","props":{"text":"Employees","variant":"large-heavy"}}},{"component":{"component":"Table","props":{"columns":[{"key":"name","label":"Name","type":"string"},{"key":"role","label":"Role","type":"string"}],"rows":[{"name":"Alice","role":"Engineer"},{"name":"Bob","role":"Designer"}]}}}]}]}}
```

**357 tokens** — the highest in this comparison. The nested `"component"` wrappers create deep nesting that compounds the token cost, especially for complex layouts.

### 4. OpenUI Lang (The DSL)

OpenUI Lang is a purpose-built, line-oriented domain-specific language for expressing UI component trees.

```text
root = Stack([title, tbl])
title = TextContent("Employees", "large-heavy")
tbl = Table(cols, rows)
cols = [Col("Name", "string"), Col("Role", "string")]
rows = [{name: "Alice", role: "Engineer"}, {name: "Bob", role: "Designer"}]
```

**148 tokens**. More than 50% fewer tokens than any other format. The savings come from deliberate design choices we'll unpack in a later section.

The benchmark covers seven real-world UI scenarios, from a simple table to a full e-commerce product page. Here are the results:

| Scenario | YAML | Vercel JSON-Render | Thesys C1 JSON | OpenUI Lang | vs YAML | vs Vercel | vs C1 |
|---|---|---|---|---|---|---|---|
| simple-table | 316 | 340 | 357 | **148** | -53.2% | -56.5% | -58.5% |
| chart-with-data | 464 | 520 | 516 | **231** | -50.2% | -55.6% | -55.2% |
| contact-form | 762 | 893 | 849 | **294** | -61.4% | -67.1% | -65.4% |
| dashboard | 2,128 | 2,247 | 2,261 | **1,226** | -42.4% | -45.4% | -45.8% |
| pricing-page | 2,230 | 2,487 | 2,379 | **1,195** | -46.4% | -52.0% | -49.8% |
| settings-panel | 1,077 | 1,244 | 1,205 | **540** | -49.9% | -56.6% | -55.2% |
| e-commerce-product | 2,145 | 2,449 | 2,381 | **1,166** | -45.6% | -52.4% | -51.0% |
| **TOTAL** | **9,122** | **10,180** | **9,948** | **4,800** | **-47.4%** | **-52.8%** | **-51.7%** |

### Key Takeaways

**The savings are consistent, not cherry-picked.** Across all seven scenarios — from a 148-token simple table to a 1,226-token dashboard — OpenUI Lang delivers between 42% and 67% token reduction. There's no scenario where another format comes close.

**The gap widens for form-heavy UI.** The contact-form scenario shows the largest absolute and relative savings (-67.1% vs Vercel, -61.4% vs YAML). Forms have many components with similar structures, which is exactly where OpenUI Lang's variable references and positional arguments shine.

**JSON formats are consistently more expensive than YAML.** Counterintuitively, YAML outperforms both JSON-based formats in every scenario. JSON's braces, quotes, and key duplication add up fast. Vercel's operational overhead and C1's component wrappers compound this further.

## Why OpenUI Lang Wins: Four Design Principles

The token savings aren't accidental. They're the result of four deliberate design decisions that target the specific ways LLMs waste tokens on UI serialization.

### Principle 1: Positional Arguments Replace Key-Value Pairs

In JSON and YAML, every property requires both a key and a value: `"text": "Employees"`, `"type": "Table"`, `"variant": "large-heavy"`. For a component with five props, that's five keys the LLM must generate — tokens that convey no semantic information because the schema already defines what each position means.

OpenUI Lang maps positional arguments to named props via a schema file. The LLM writes `TextContent("Employees", "large-heavy")` instead of `{"type":"TextContent","props":{"text":"Employees","variant":"large-heavy"}}`. The parser resolves `"Employees"` to the `text` prop and `"large-heavy"` to the `variant` prop using the component's schema definition.

This single decision eliminates roughly 30-40% of the token overhead in most components.

### Principle 2: Line-Oriented Structure Eliminates Nesting

JSON and YAML use nested braces or indentation to express hierarchy. Every level of nesting requires opening and closing delimiters, and in JSON's case, those delimiters must be on the same or adjacent lines, which the LLM must track and balance.

OpenUI Lang uses variable references instead. A child component is defined as a named variable and referenced by name in the parent:

```text
root = Stack([title, tbl])
title = TextContent("Employees", "large-heavy")
tbl = Table(cols, rows)
```

No nested braces. No indentation tracking. The LLM generates flat lines, and the parser reconstructs the tree from the references. This is not just more token-efficient — it's also easier for the LLM to generate correctly, reducing structural errors.

### Principle 3: Variable References Enable Deduplication

In JSON/YAML formats, if you reference the same data in multiple places (e.g., a list of columns used in both a table and a chart), you must duplicate it. Each duplication is a full copy of the tokens.

OpenUI Lang lets you define data once and reference it by name:

```text
cols = [Col("Name", "string"), Col("Role", "string"), Col("Salary", "currency")]
tbl = Table(cols, rows)
chart = BarChart(cols, rows)
```

The `cols` variable is defined once (15 tokens) and used twice (2 tokens per reference). In JSON, you'd emit the full column definitions twice — roughly 60 additional tokens.

### Principle 4: Schema-Driven Parsing Shifts Complexity to the Machine

The key insight is that **the LLM doesn't need to be the serializer**. In JSON-based approaches, the LLM must produce valid JSON with correct nesting, quoting, and key names. This is error-prone and token-expensive.

OpenUI Lang shifts the structural complexity to the schema and parser. The LLM produces a simplified, line-oriented representation. The parser — running on cheap, deterministic hardware — resolves the schema, validates types, and builds the full component tree. The LLM does less work per output token, and each token carries more semantic weight.

## The Real Cost: Dollars at Scale

Tokens are abstract. Dollars are not. Let's project these savings across realistic production volumes.

### Assumptions

- **Average tokens per response** (output): 686 tokens (benchmark total ÷ 7 scenarios)
- **Average OpenUI Lang tokens per response**: 4,800 ÷ 7 = 686 tokens
- **Average JSON/YAML tokens per response**: 9,775 ÷ 7 ≈ 1,396 tokens (average of YAML, Vercel, C1)
- **GPT-5.2 output pricing**: $15 per 1M output tokens (approximate; adjust to current rates)
- **GPT-5.2 input pricing**: $7.50 per 1M input tokens (for multi-turn context)
- **Average conversation length**: 3 turns (so output tokens also become input tokens in subsequent turns)

### Monthly Cost Projection

| Scale (requests/month) | JSON/YAML Output Cost | OpenUI Lang Output Cost | Monthly Savings | Annual Savings |
|---|---|---|---|---|
| 100K | $209 | $103 | $106 | $1,272 |
| 500K | $1,047 | $514 | $533 | $6,396 |
| 1M | $2,094 | $1,029 | $1,065 | $12,780 |
| 5M | $10,472 | $5,143 | $5,329 | $63,948 |
| 10M | $20,944 | $10,286 | $10,658 | $127,896 |

### The Compounding Effect of Multi-Turn Conversations

Output tokens don't disappear — they become input tokens in the next turn. In a 3-turn conversation, each generated response is fed back as context. That means:

- **JSON/YAML**: 1,396 output tokens × 2 additional turns as input = 2,792 extra input tokens per conversation
- **OpenUI Lang**: 686 output tokens × 2 additional turns as input = 1,372 extra input tokens per conversation

At 1M conversations/month, the input-cost difference alone is roughly $1,065/month — effectively doubling the savings. For teams building conversational AI with persistent context, this compounding is the real budget killer.

### Latency: Time Is Also Money

Token count directly affects time-to-first-byte and total render latency. At 60 tokens/second generation speed:

| Scenario | JSON/YAML Tokens | OpenUI Lang Tokens | Time Saved |
|---|---|---|---|
| simple-table | 316 | 148 | 2.8 seconds |
| contact-form | 762 | 294 | 7.8 seconds |
| dashboard | 2,128 | 1,226 | 15.0 seconds |

The contact-form scenario renders **3.04× faster** with OpenUI Lang. The simple-table renders **2.30× faster**. For user-facing applications, these latency differences directly impact engagement and conversion. A dashboard that renders 15 seconds faster isn't just cheaper — it's a fundamentally better user experience.

## The Honest Tradeoffs

Token efficiency isn't everything. Here's what you give up with each approach, and when the "less efficient" option might still be the right call.

### Where OpenUI Lang Falls Short

**Ecosystem immaturity.** YAML and JSON have decades of tooling — linters, validators, IDE support, formatters, diff tools. OpenUI Lang is a DSL, which means you're investing in a custom parser, custom tooling, and a smaller community. If your team already has robust YAML/JSON pipelines, the switching cost is real.

**Debugging opacity.** When a JSON response breaks, you can paste it into any JSON validator and get a line number. When a DSL response breaks, you need the OpenUI Lang parser to tell you what went wrong. Debugging tooling for DSLs is inherently less mature.

**Schema coupling.** OpenUI Lang's positional argument system requires a schema that maps positions to prop names. If your component API changes frequently, you're updating both your code and your schema. JSON/YAML are self-describing — the key names travel with the data.

**Limited interoperability.** If you need to pass your UI descriptions to external systems, APIs, or databases, JSON is the lingua franca. You'd need to compile OpenUI Lang to JSON for storage or transmission, adding a build step.

### Where JSON/YAML Still Win

**Tool integration.** If your Generative UI pipeline feeds into existing systems — CMS platforms, API gateways, analytics pipelines — JSON is the path of least resistance. No translation layer needed.

**Streaming semantics.** Vercel's JSON Patch format was designed for streaming. Each `op: "add"` line is an independent, parseable unit. You can render elements as they arrive without waiting for the full response. OpenUI Lang can support streaming (it's line-oriented, after all), but the streaming story is less mature.

**Familiarity.** Every developer knows JSON. Most know YAML. Onboarding new engineers to a custom DSL has a real, measurable cost in ramp-up time and cognitive overhead.

**Debugging and observability.** JSON structures are trivially loggable, queryable, and inspectable. Your existing observability stack already handles JSON. A DSL introduces a format your logging and monitoring tools may not understand natively.

### The Spectrum of Choices

This isn't a binary decision. Practical deployments often combine approaches:

- **Generate in OpenUI Lang, store as JSON.** Get the token savings at generation time, the interoperability for persistence.
- **Use YAML for simple cases, OpenUI Lang for complex ones.** A hybrid parser that handles both formats lets you optimize where it matters.
- **Prototype in JSON, optimize to OpenUI Lang later.** Start with familiar tooling, migrate the hot path to the DSL once you understand your actual workload.

## The Bigger Picture: Why This Matters Now

Generative UI is moving from demos to production. The difference between a prototype that costs $50/month and a production system that costs $5,000/month is often just scale — and token count is the multiplier.

But there's a deeper trend: **the intelligence-to-syntax ratio**. As models get smarter, the value they provide per token increases. The faster we can strip structural syntax from the output and let the model focus on semantic decisions — *what* component to use, *what* data to show, *what* layout to choose — the more efficient the entire system becomes.

OpenUI Lang represents one approach to this problem: a purpose-built DSL that minimizes the tokens spent on mechanical serialization and maximizes the tokens spent on actual UI intelligence. The 47-53% reduction in token count isn't just a cost optimization — it's a signal that the industry is moving toward formats designed for LLMs rather than formats inherited from human configuration workflows.

The question isn't whether purpose-built DSLs will become standard for Generative UI. The question is which ones will win, and what tradeoffs they'll make to get there.

## Conclusion

The data is clear:

- **OpenUI Lang uses 47-53% fewer tokens** than YAML, Vercel AI SDK, and C1 JSON across all tested scenarios.
- **At 1M requests/month**, that translates to roughly **$1,065/month in direct output cost savings**, with additional savings from reduced input token compounding in multi-turn conversations.
- **Latency improvements of 2-3×** directly impact user experience for complex UI generation.
- **The savings come from four design principles**: positional arguments, line-oriented structure, variable references, and schema-driven parsing.

But token efficiency alone doesn't make a framework the right choice. YAML's ubiquity, JSON's interoperability, and Vercel's streaming semantics are real advantages that matter in production systems. The right choice depends on your scale, your team's expertise, your integration requirements, and how much of your budget is actually consumed by token costs versus engineering time.

For teams where token costs are a material line item — and if you're reading this, they probably are — OpenUI Lang offers a compelling answer to a question the industry is only starting to take seriously: *how much are you paying just to format your AI's output?*
